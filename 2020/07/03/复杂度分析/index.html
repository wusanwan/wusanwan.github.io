算法：是解决问题的有效步骤。

频度：该语句在算法中被重复执行的次数。

T(n)：算法中所有语句的频度之和。

S(n): 代码的空间复杂度。

🥀复杂度分析的符号

O(f(n)) 大O T(n)的增长率小于等于c (f(n)) *上界

Ω(g(n)) sigma T(n)的增长率大于等于c (g(n)) *下界

Θ(h(n)) theta T(n)的增长率等于c*(h(n))

o(p(n)) 小o T(n)的增长率小于c*(p(n))

如果f(n)=n^2的话 f(n)=O(n^2),或f(n)=O(n^3)(就数学定义而言)都是对的，考研以及生活中使用而言后者是错误的(感觉已经引申为最小上界的意思)。 但Θ的话就要尽可能严密，用f(n)=Θ(n^2)

Q:为什么平时都用大O ，而非Θ？

A:因为代码的复杂度不是固定不变的，取决于问题的规模以及输入的数据。大O是上界表示这段代码最悲观的情况，比如说所有循环都算上的情况，Ω表示最乐观的情况，可能连循环都没进，Θ表示最乐观和最悲观情况相同，不因输入而改变。一方面这种代码比较少，Θ没有用武之地，另一方面工程师更倾向于对程序运行时的复杂程度作最坏估计，以便于进行改进。（顺便提一下平均复杂度，平均复杂度大多很难计算。）

☄时间复杂度分析的法则

数学上的法则：

T1(N)=O(f(N))且T2(N)=O(g(N))

  1. T1(N)+T2(N)=max(O(f(N),O(g(N))
  2. T1(N)T2(N)=O(f(N)g(N))

可以通过计算极限来比较函数的相对增长率。

程序上的法则：

1.循环语句，根据上面第2条，k重循环就是O（nnnn…n）=O(n^k)

2.顺序语句，即为上面T1(N)+T2(N)的情况，max(O(f(N),O(g(N)))

3.选择语句，max(T1(N),T2(N))的情况，max(O(f(N),O(g(N)))

时间复杂度分析快速入门：题型分类法

☄时间复杂度分析的顺序

基本策略：从最深层自内向外展开。

1.函数调用，最先分析。

2.递归过程，有几种情况。

a.本质是for循环的递归。

b.其他递归（比如斐波那契数列的递归，有大量重复计算）

分析递归的策略

  1. 分析整体的流程写一个T(n)=f(T(n),O(f(n))的表达式
  2. 将O(f(n))近似为f(n)

  3. 时间复杂度=迭代次数T1(n)*每次迭代的复杂度T2(n) 法则2

    迭代次数即为从n到基准情况要迭代几次(具体分析可见下面的ex4)

时间复杂度的大小：O(1) < O(logn) < O(n) < O(nlogn) < O(n^2) < O(n^3) < O(2^n) < O(n!) < O(n^k)

技巧：如果在O(1)时间内把问题一削就削1/n那这个部分就是O(logn)(比如二分查找)，如果只削了常数规模的话就是O(n）

    品一品同与不同。

    设执行次数为t,有2^t<n，即t<=log2n ,故T(n)=O(logn)

    设执行次数为t，有t(t+1)/2=n,反解可知T(n)=O(n^(1/2))

    设执行次数为t，有t^2<n，即t<sqrt(n),故T(n)=O(sqrt(n))

    设执行次数为t，有i^(2^(t+1))<n，即t<loglogn-1，故T(n)=O(loglogn)

  • 🍵例5:链表合并

    已知两个长度分别为m和n的升序链表，若将它们合并为长度为m+n的一个降序链表，则最坏情况下的时间复杂度是？

    A. O(n) B.O(mn) C.O(nlogn) D.O(max(m,n))

    D,两个链表的元素依次进行比较，再头插法。

    此题解析

    Q:为什么不是O(m+n)

    A：看那个代码因为是顺序结构的所以就是法则1max(O(f(N),O(g(N))

    max(m,n)就是对应的代码的“级数”层次，而m+n对应的是“实际执行次数“（没有用法则对顺序结构取max，而取加号）

    脑子里想的时候一定要小心，不是实际执行次数，而是级数层次，如果是给你实际的代码让你分析可能不会分析错。

  • ✍例6:规范化的表达

    首先设执行次数为t，有(t+5+1)(t+5+1)<n 得t<sqrt(n)-6,然后里面又是O(1),就有T(n)=O(1sqrt(n))

  • ✍例7:你以为你会了的考研例子4:斐波那契数列递归和非递归版分析

    1,1，2,3，5,8，13,21

    非递归版很简单，就是循环O(n)

    递归版

    ☹错误示范：从F(n)=F(n-1)+F(n-2)

    可知F(n)是每次减1，执行次数是O(n),调用的话是O(1），所以整体是O(n).

    之前为什么可以从n的变化来估计执行次数呢，因为人家只调用自身一次，等于是循环了。

    现在就是每次要调用自身2次。（其实调用自身以后也会继续递归下去）

    😎正确示范：

    F(n) 调用2次，F(n-1)调用2次，……难道是O(2*n)=O(n)吗，你又错了。

    它这里的F(x)不是一次调用就结束的，而是在多个不同的地方被重复调用。

    而且它调用完又不是停了，而是是继续调用自己走到底，

    产生了大量重复计算，这就是为啥斐波那契数列的递归计算非常费时。

    F(n)不止调用2次，F(n）调用2^n次，F(n)调用F(n-1),F(n-2),F(n-1)又调用F(n-2)，F(n-3)……

    就是一个类似于二叉树一样的东西，所以结果是O（2^n*1）=O(2^n)。

  1. 当n翻倍时，你的算法运行时间扩大了几倍？
  2. 计算T(n)/f(n)，看是否收敛于非0的正常值。其中T(n)是凭经验的时间，f(n)是运行出来的近似时间。

🛑空间复杂度

空间复杂度S(n)定义为算法所耗费的存储空间，是问题规模n的函数。S(n)=O(g(n))

原地工作：算法所需的辅助空间是常量，即O(1）不是说只需要1/0个单位的空间
